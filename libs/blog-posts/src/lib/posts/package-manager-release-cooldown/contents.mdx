# Configuring minimum release age across npm, pnpm, yarn, and bun

A minimum release age (or "cooldown") instructs a package manager to ignore any dependency version published less than a configured amount of time ago. Most malicious releases are reported and removed from the registry within a few hours of publication, so even a 24-hour delay screens out the majority of compromised versions before they reach a local install.<Cite n={1} href="https://gist.github.com/mcollina/b294a6c39ee700d24073c0e5a4e93104">{collinaGist}</Cite>

Each major Node.js package manager now supports this configuration, though the setting name and the unit of time differ in each one. The reference below was assembled from Matteo Collina's original gist covering npm, pnpm, and Yarn,<Cite n={1} href="https://gist.github.com/mcollina/b294a6c39ee700d24073c0e5a4e93104">{collinaGist}</Cite> Tom Ricci's expansion adding Bun,<Cite n={2} href="https://gist.github.com/tom-ricci/bade55606ebb7a550598f11e29f1d6b0">{ricciGist}</Cite> and the official documentation for each tool.<Cite n={3} href="https://docs.npmjs.com/cli/v11/using-npm/config#min-release-age">{npmDocs}</Cite><Cite n={4} href="https://pnpm.io/settings#minimumreleaseage">{pnpmDocs}</Cite><Cite n={5} href="https://yarnpkg.com/configuration/yarnrc#npmMinimalAgeGate">{yarnDocs}</Cite><Cite n={6} href="https://bun.sh/docs/runtime/bunfig#install-minimumreleaseage">{bunDocs}</Cite>

## Table of Contents

## Quick reference

- **npm**: `min-release-age`, in **days**. Supported in npm CLI **11.10.0** (February 2026).<Cite n={3} href="https://docs.npmjs.com/cli/v11/using-npm/config#min-release-age">{npmDocs}</Cite>
- **pnpm**: `minimumReleaseAge`, in **minutes**. Supported in pnpm **10.16** (September 2025). The default became `1440` in pnpm **11.0**.<Cite n={4} href="https://pnpm.io/settings#minimumreleaseage">{pnpmDocs}</Cite>
- **Yarn**: `npmMinimalAgeGate`, in **minutes**. Supported in Yarn Berry **4.10.0**.<Cite n={5} href="https://yarnpkg.com/configuration/yarnrc#npmMinimalAgeGate">{yarnDocs}</Cite>
- **Bun**: `minimumReleaseAge`, in **seconds**. Supported in Bun **1.3.0**.<Cite n={6} href="https://bun.sh/docs/runtime/bunfig#install-minimumreleaseage">{bunDocs}</Cite>

## Caveats

Several aspects of the cooldown's behavior can cause it to be silently bypassed. The pitfalls below are grouped by which configuration scope they affect; each tends to bite one form of setup more than the other.

### Per-project pitfalls

#### The check runs at install time

The cooldown is enforced when the package manager resolves and installs a version, not when an automation suggests one. Renovate and Dependabot will open update PRs regardless of any cooldown configured in the package manager itself, so their own cooldown options must be set independently. Renovate 42's `config:best-practices` preset enables a three-day cooldown for npm by default.<Cite n={1} href="https://gist.github.com/mcollina/b294a6c39ee700d24073c0e5a4e93104">{collinaGist}</Cite> Dependabot exposes the equivalent control as `cooldown.default-days`.

#### No per-registry scope

None of the four tools support applying the gate to one registry but not another. Internal packages from a private registry are subject to the cooldown unless explicitly listed in the per-tool exclusion option. npm does not yet expose an exclusion option; an issue is open against the CLI tracking it.<Cite n={7} href="https://github.com/npm/cli/issues/8994">{npmExcludeIssue}</Cite>

#### Defaults are likely to shift

pnpm 11 enables the gate by default at 1440 minutes, and npm CLI v12 is expected to do the same.<Cite n={1} href="https://gist.github.com/mcollina/b294a6c39ee700d24073c0e5a4e93104">{collinaGist}</Cite> Setting the value explicitly today preserves the current configured behavior when the upstream defaults change.

### User-wide pitfalls

#### A repository pinning an older package manager disables the gate

If a project pins its package manager via the `packageManager` field in `package.json`, Corepack, Volta, a Docker base image, or a CI action such as `setup-node` with `version-file`, the pinned version is the one that performs installs. When the pinned version predates cooldown support, no user-wide setting has any effect inside that repository; an older binary has no concept of the option and ignores it.

This is the most common reason a user-wide cooldown is configured but not active. The version that actually performs the install can be confirmed by running `pnpm --version` (or the equivalent for npm, yarn, or bun) from inside the repository, or by inspecting the `setup-node` step in the CI logs.

#### A repository's looser configuration overrides a stricter user configuration

Project-level configuration takes precedence over user-level configuration in all four tools. A user-wide cooldown of one day does not apply inside a repository whose `.npmrc`, `pnpm-workspace.yaml`, `.yarnrc.yml`, or `bunfig.toml` sets a smaller value or no value at all. The setting behaves as project policy rather than as a per-user safety preference. A user-wide value, therefore, cannot serve as a backstop for repositories that have not opted in.

## Per-project configuration

Per-project configuration is committed to the repository and applies to every developer and CI run that uses a supported package manager version. The selected tab is remembered across the rest of the post.

<Tabs groupId="package-manager">
<Tab label="npm">

The setting belongs in the project's `.npmrc`:

```INI filename=".npmrc"
min-release-age=1
```

The value is in days. The setting cannot be combined with `--before` in the same invocation; npm raises an error if both are provided. An npm CLI version that does not recognize the key emits the following warning, which is cosmetic and does not prevent the setting from applying on a supporting CLI:

```text
npm warn Unknown project config "min-release-age". This will stop working in the next major version of npm.
```

</Tab>
<Tab label="pnpm">

Since pnpm 11, the canonical location is `pnpm-workspace.yaml`:

```yaml filename="pnpm-workspace.yaml"
minimumReleaseAge: 1440
```

Internal packages that should bypass the gate can be listed under `minimumReleaseAgeExclude`:

```yaml filename="pnpm-workspace.yaml"
minimumReleaseAge: 1440
minimumReleaseAgeExclude:
  - '@platformatic/*'
```

On pnpm 10.x, the same setting is written in `.npmrc` with a different casing:

```INI filename=".npmrc"
minimum-release-age=1440
```

A known bug on pnpm 10.x causes `minimumReleaseAge` to be silently ignored when any package in the workspace sets `shared-workspace-lockfile=false` in its `.npmrc`.<Cite n={8} href="https://github.com/pnpm/pnpm/issues/10008">{pnpmLockfileBug}</Cite> On 10.x, this is worth verifying before assuming the setting is honored.

</Tab>
<Tab label="Yarn">

The setting belongs in `.yarnrc.yml` (Yarn Berry 4.10 or later):

```yaml filename=".yarnrc.yml"
npmMinimalAgeGate: 1440
```

Packages that should bypass the gate can be listed under `npmPreapprovedPackages`:

```yaml filename=".yarnrc.yml"
npmMinimalAgeGate: 1440
npmPreapprovedPackages:
  - '@platformatic/*'
```

`npmPreapprovedPackages` accepts both glob patterns and exact locators (for example, `@aws-sdk/types@3.877.0`), giving it more granularity than pnpm's package-name-only exclude list.

The documentation states that a duration suffix such as `"7d"` is accepted, but a known parsing bug causes the suffix to be dropped.<Cite n={9} href="https://github.com/yarnpkg/berry/issues/6991">{yarnSuffixBug}</Cite> Until that is resolved, a plain minute count is the reliable form: one day is `npmMinimalAgeGate: 1440`.

</Tab>
<Tab label="Bun">

The setting belongs under `[install]` in `bunfig.toml`:

```toml filename="bunfig.toml"
[install]
minimumReleaseAge = 86400
```

The value is in seconds. One day is `86400`.

Packages that should bypass the gate can be listed under `minimumReleaseAgeExcludes`:

```toml filename="bunfig.toml"
[install]
minimumReleaseAge = 86400
minimumReleaseAgeExcludes = ["@types/node", "typescript"]
```

</Tab>
</Tabs>

## User-wide configuration

User-wide configuration applies on a single machine, across every repository that does not override it. As noted in the user-wide pitfalls above, project-level configuration takes precedence over the values shown here.

<Tabs groupId="package-manager">
<Tab label="npm">

```bash
npm config set min-release-age 1 --location=user
```

This writes to `~/.npmrc`. Substituting `global` for `user` writes to the global npm configuration instead.

</Tab>
<Tab label="pnpm">

```bash
pnpm config set minimumReleaseAge 1440 --location=user
```

On pnpm 11 and later, non-auth settings written this way are stored in `~/.config/pnpm/config.yaml` rather than `~/.npmrc`.

</Tab>
<Tab label="Yarn">

```bash
yarn config set --home npmMinimalAgeGate 1440
```

`--home` writes to `~/.yarnrc.yml` rather than the project-local file.

</Tab>
<Tab label="Bun">

Bun reads project-level `bunfig.toml` only; user-wide configuration is not yet supported. An open pull request adds the corresponding behavior.<Cite n={10} href="https://github.com/oven-sh/bun/pull/28727">{bunUserWidePr}</Cite> Once merged, the same key written in a user-wide `bunfig.toml` will apply across projects.

</Tab>
</Tabs>
