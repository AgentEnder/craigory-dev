Configuring minimum release age across npm, pnpm, yarn, and bun

May 29, 2026

A minimum release age — or “cooldown” — tells your package manager to ignore any version of a dependency that was published less than N time ago. It is a one-line config change that buys you most of the protection you would get from a sophisticated supply-chain monitoring setup: the typical malicious release is caught and pulled from the registry within a few hours, so even a one-day delay routes around the worst of them.[1]

Every major Node.js package manager now supports this, but each picked its own name for the setting and its own unit of time. The notes below are mostly a cross-reference of what to write, where to write it, and the parts that bit me (or are about to). The technical bones come from Matteo Collina’s original gist on the npm / pnpm / Yarn setup,[1] Tom Ricci’s expansion that added Bun,[2] and each tool’s own documentation.[3][4][5][6]

Quick reference
#

  • npm — min-release-age, measured in days, available from npm CLI 11.10.0 (Feb 2026).[3]
  • pnpm — minimumReleaseAge, measured in minutes, available from pnpm 10.16 (Sep 2025); the default became 1440 in pnpm 11.0.[4]
  • Yarn — npmMinimalAgeGate, measured in minutes, available from Yarn Berry 4.10.0.[5]
  • Bun — minimumReleaseAge, measured in seconds, available from Bun 1.3.0.[6]

Things to know before you turn it on
#

I have these up front because three of them are how you think the cooldown is protecting you when it actually isn’t.

The gate runs at install, not when a bot opens a PR
#

The check happens when the package manager resolves and installs a version. A bot that proposes an update is doing neither — so Renovate and Dependabot will happily file PRs against versions your install command will refuse. Configure their cooldown options too: Renovate 42’s config:best-practices preset already enables a three-day cooldown for npm,[1] and Dependabot’s cooldown.default-days is the corresponding knob.

A repo that pins an older package manager wins, every time
#

If a project pins its package manager — via the packageManager field in package.json, Corepack, Volta, a Docker base image, the setup-node action’s version-file input, anything — the version the project pins is the version that actually runs your install. If that pinned version predates cooldown support, the gate does not exist for that repo, no matter what your globally installed package manager is or what your user-level config says. An older binary has no concept of the setting and silently ignores it.

This is the failure mode I see most often. The fix is to confirm the version that actually runs (pnpm --version from inside the repo, corepack pnpm --version, the setup-node log line in CI) is new enough to honor the setting.

A lax repo config beats a stricter user config
#

Per-project settings take precedence over per-user settings in all four tools. If your ~/.npmrc / ~/.config/pnpm/config.yaml / ~/.yarnrc.yml / ~/.bunfig.toml sets a one-day cooldown but the project sets zero — or sets nothing on an older tool version where the default is zero — the project value wins. The setting behaves like a project policy, not a personal safety preference.

Practically, this means setting cooldown globally on your laptop is not a substitute for setting it in each repo. A repo that has not opted in is a repo without a cooldown, for you.

No per-registry scope
#

None of the four tools let you apply the gate to one registry but not another. Internal packages from a private registry get delayed alongside everything else unless you list them in the per-tool exclude option. npm does not have an exclude option yet — there is an open issue tracking it.[7]

Defaults are about to move
#

pnpm 11 already enables the gate at 1440 minutes by default; npm CLI v12 is expected to follow.[1] If you set a value explicitly today, you keep the behavior you wrote down when the defaults shift under you.

npm
#

Per-project
#

Put the setting in the project’s .npmrc:

.npmrc
min-release-age=1

The value is in days. You cannot combine it with --before in the same invocation; npm errors out if both are present. Older npm CLI versions that don’t recognize the key will print:

npm warn Unknown project config "min-release-age". This will stop working in the next major version of npm.

The warning is cosmetic — the setting still applies on a CLI that supports it, and is harmlessly ignored on one that doesn’t.

User-wide
#

npm config set min-release-age 1 --location=user

That writes to ~/.npmrc. Swap user for global if you want it in the global npm config instead.

pnpm
#

Per-project
#

Since pnpm 11, the canonical home is pnpm-workspace.yaml:

pnpm-workspace.yaml
minimumReleaseAge: 1440

If you publish internal packages and want them to bypass the gate:

pnpm-workspace.yaml
minimumReleaseAge: 1440
minimumReleaseAgeExclude:
  - '@platformatic/*'

On pnpm 10.x you write the same idea in .npmrc with a different casing:

.npmrc
minimum-release-age=1440

Worth knowing on 10.x: there is an open bug where minimumReleaseAge is silently ignored if any package in the workspace has shared-workspace-lockfile=false set in its .npmrc.[8] If you’ve configured the gate but it doesn’t seem to be doing anything, that’s the first thing I’d check.

User-wide
#

pnpm config set minimumReleaseAge 1440 --location=user

On pnpm 11+, non-auth settings written this way land in ~/.config/pnpm/config.yaml rather than ~/.npmrc.

Yarn (Berry 4.10+)
#

Per-project
#

In .yarnrc.yml:

.yarnrc.yml
npmMinimalAgeGate: 1440

If some packages need to skip the gate:

.yarnrc.yml
npmMinimalAgeGate: 1440
npmPreapprovedPackages:
  - '@platformatic/*'

npmPreapprovedPackages accepts both glob patterns and exact locators (e.g. @aws-sdk/types@3.877.0), which is a notch more flexible than pnpm’s package-name-only exclude list.

The docs suggest you can pass a duration string like "7d" instead of a number, but there’s an outstanding bug where the suffix is silently dropped.[9] Until that’s resolved everywhere, stick to a plain minute count: one day is npmMinimalAgeGate: 1440.

User-wide
#

yarn config set --home npmMinimalAgeGate 1440

--home writes to ~/.yarnrc.yml instead of the project-local file.

Bun
#

Per-project
#

Bun puts it under [install] in bunfig.toml:

bunfig.toml
[install]
minimumReleaseAge = 86400

The value is in seconds. One day is 86400.

To skip the gate for a few packages, use minimumReleaseAgeExcludes:

bunfig.toml
[install]
minimumReleaseAge = 86400
minimumReleaseAgeExcludes = ["@types/node", "typescript"]

User-wide
#

There is no user-wide config story yet — Bun reads project-level bunfig.toml only. An open PR is wiring up support;[10] when it merges, the same key in a user-wide bunfig.toml will work the way it does for the other tools.
