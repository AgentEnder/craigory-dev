Configuring minimum release age across npm, pnpm, yarn, and bun

May 29, 2026

A minimum release age (or “cooldown”) instructs a package manager to ignore any dependency version published less than a configured amount of time ago. The reference below was assembled from Matteo Collina’s original gist covering npm, pnpm, and Yarn,[1]Matteo Collina, "Configuring minimum release age across npm, pnpm, and yarn" (gist, April 2026). Overview
#

What the setting does
#

A cooldown delays the moment at which a freshly published version becomes installable. With a one-day cooldown, a version published less than 24 hours ago is invisible to the resolver: installs proceed as if it did not exist yet.

The value of this gate comes from the typical lifecycle of a malicious release. Compromised versions are usually flagged and removed from the registry within a few hours of publication, often before most teams have seen them.[1]Matteo Collina, "Configuring minimum release age across npm, pnpm, and yarn" (gist, April 2026). Original write-up covering npm, pnpm, and Yarn. A 24-hour delay is enough to filter the majority of those incidents at the install layer without requiring any monitoring, scanning, or human review.

The gate does not detect malicious code, audit dependencies, or block known-bad versions. It is a single, blunt rule applied to every package version in the resolution: too new, ignored. That bluntness is the source of its leverage; it costs effectively nothing to turn on and runs on every install without configuration drift.

When each tool added support
#

Each tool picked its own setting name and its own unit of time:

  • npm: min-release-age, in days. npm CLI 11.10.0, released February 2026.[3]npm CLI documentation, min-release-age.
  • pnpm: minimumReleaseAge, in minutes. pnpm 10.16, released September 2025; the default became 1440 in pnpm 11.0.[4]pnpm documentation, minimumReleaseAge.
  • Yarn: npmMinimalAgeGate, in minutes. Yarn Berry 4.10.0.[5]Yarn documentation, npmMinimalAgeGate.
  • Bun: minimumReleaseAge, in seconds. Bun 1.3.0.[6]Bun documentation, install.minimumReleaseAge.

Why configure both scopes
#

The setting can be written into a repository’s package-manager config or into the user-level config on a machine. The two scopes answer different questions, and neither replaces the other.

A per-project value is committed to the repository. It applies to every developer and every CI run that uses a supported package manager version. It is the canonical place to declare policy for a codebase, and the only way to ensure all contributors observe the same cooldown without coordinating their dotfiles. It does nothing in repositories that have not been configured.

A user-wide value applies on a single machine, across every repository whose project-level config does not override it. It is the only way to extend the cooldown to repositories that have not opted in, including dependencies cloned for inspection, scratch projects, and other people’s repos opened from a clone. It is silently bypassed inside repositories that pin an older package manager, and inside repositories that declare a looser value at the project level (see the caveats under each section).

The two scopes compose. A per-project value protects the codebase against published policy violations; a user-wide value protects the machine against repositories that have not yet taken responsibility for their own cooldown. Setting both is the only configuration that gives a known floor everywhere the machine touches.

Per-project
#

Setup
#

Per-project configuration lives in the package manager’s project-local config file and is committed to the repository. The selected tab is remembered across the rest of the post.

The setting belongs in the project’s .npmrc:

.npmrc
min-release-age=1

The value is in days. The setting cannot be combined with --before in the same invocation; npm raises an error if both are provided. An npm CLI version that does not recognize the key emits the following warning, which is cosmetic and does not prevent the setting from applying on a supporting CLI:

npm warn Unknown project config "min-release-age". This will stop working in the next major version of npm.

Since pnpm 11, the canonical location is pnpm-workspace.yaml:

pnpm-workspace.yaml
minimumReleaseAge: 1440

Internal packages that should bypass the gate can be listed under minimumReleaseAgeExclude:

pnpm-workspace.yaml
minimumReleaseAge: 1440
minimumReleaseAgeExclude:
  - '@platformatic/*'

On pnpm 10.x, the same setting is written in .npmrc with a different casing:

.npmrc
minimum-release-age=1440

A known bug on pnpm 10.x causes minimumReleaseAge to be silently ignored when any package in the workspace sets shared-workspace-lockfile=false in its .npmrc.[8]pnpm issue #10008. minimumReleaseAge silently ignored on pnpm 10.x when any workspace package sets shared-workspace-lockfile=false. On 10.x, this is worth verifying before assuming the setting is honored.

The setting belongs in .yarnrc.yml (Yarn Berry 4.10 or later):

.yarnrc.yml
npmMinimalAgeGate: 1440

Packages that should bypass the gate can be listed under npmPreapprovedPackages:

.yarnrc.yml
npmMinimalAgeGate: 1440
npmPreapprovedPackages:
  - '@platformatic/*'

npmPreapprovedPackages accepts both glob patterns and exact locators (for example, @aws-sdk/types@3.877.0), giving it more granularity than pnpm’s package-name-only exclude list.

The documentation states that a duration suffix such as "7d" is accepted, but a known parsing bug causes the suffix to be dropped.[9]Yarn issue #6991. Duration suffixes silently dropped when parsing npmMinimalAgeGate. Until that is resolved, a plain minute count is the reliable form: one day is npmMinimalAgeGate: 1440.

The setting belongs under [install] in bunfig.toml:

bunfig.toml
[install]
minimumReleaseAge = 86400

The value is in seconds. One day is 86400.

Packages that should bypass the gate can be listed under minimumReleaseAgeExcludes:

bunfig.toml
[install]
minimumReleaseAge = 86400
minimumReleaseAgeExcludes = ["@types/node", "typescript"]

Caveats
#

The check runs at install time
#

The cooldown is enforced when the package manager resolves and installs a version, not when an automation suggests one. Renovate and Dependabot will open update PRs regardless of any cooldown configured in the package manager itself, so their own cooldown options must be set independently. Renovate 42’s config:best-practices preset enables a three-day cooldown for npm by default.[1]Matteo Collina, "Configuring minimum release age across npm, pnpm, and yarn" (gist, April 2026). Original write-up covering npm, pnpm, and Yarn. Dependabot exposes the equivalent control as cooldown.default-days.

No per-registry scope
#

None of the four tools support applying the gate to one registry but not another. Internal packages from a private registry are subject to the cooldown unless explicitly listed in the per-tool exclusion option. npm does not yet expose an exclusion option; an issue is open against the CLI tracking it.[7]npm CLI issue #8994. Tracks an exclusion list for min-release-age.

Defaults are likely to shift
#

pnpm 11 enables the gate by default at 1440 minutes, and npm CLI v12 is expected to do the same.[1]Matteo Collina, "Configuring minimum release age across npm, pnpm, and yarn" (gist, April 2026). Original write-up covering npm, pnpm, and Yarn. Setting the value explicitly today preserves the current configured behavior when the upstream defaults change.

User-wide
#

Setup
#

User-wide configuration applies on a single machine, across every repository that does not override it. As noted in the caveats below, project-level configuration takes precedence over the values shown here.

npm config set min-release-age 1 --location=user

This writes to ~/.npmrc. Substituting global for user writes to the global npm configuration instead.

pnpm config set minimumReleaseAge 1440 --location=user

On pnpm 11 and later, non-auth settings written this way are stored in ~/.config/pnpm/config.yaml rather than ~/.npmrc.

yarn config set --home npmMinimalAgeGate 1440

--home writes to ~/.yarnrc.yml rather than the project-local file.

Bun reads project-level bunfig.toml only; user-wide configuration is not yet supported. An open pull request adds the corresponding behavior.[10]Bun pull request #28727. Adds user-wide bunfig.toml support. Once merged, the same key written in a user-wide bunfig.toml will apply across projects.

Caveats
#

A repository pinning an older package manager disables the gate
#

If a project pins its package manager via the packageManager field in package.json, Corepack, Volta, a Docker base image, or a CI action such as setup-node with version-file, the pinned version is the one that performs installs. When the pinned version predates cooldown support, no user-wide setting has any effect inside that repository; an older binary has no concept of the option and ignores it.

This is the most common reason a user-wide cooldown is configured but not active. The version that actually performs the install can be confirmed by running pnpm --version (or the equivalent for npm, yarn, or bun) from inside the repository, or by inspecting the setup-node step in the CI logs.

A repository’s looser configuration overrides a stricter user configuration
#

Project-level configuration takes precedence over user-level configuration in all four tools. A user-wide cooldown of one day does not apply inside a repository whose .npmrc, pnpm-workspace.yaml, .yarnrc.yml, or bunfig.toml sets a smaller value or no value at all. The setting behaves as project policy rather than as a per-user safety preference. A user-wide value, therefore, cannot serve as a backstop for repositories that have not opted in.
